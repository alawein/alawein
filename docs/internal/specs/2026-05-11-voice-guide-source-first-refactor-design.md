# Voice guide: source-first refactor (design spec)

Refactor the unified voice guide (`Meshal-Alawein-Voice-Export.md` on Desktop) into
a doctrine-compliant, version-controlled artifact in the `alawein` repo with a
single source of truth, no Mercor/Yosemite leak, and an architecture that survives
further enhancement without merge-against-output bugs.

This spec is the output of the 2026-05-11 brainstorming session that critiqued the
prior `voice-guide-enhancement-prompt.md` plan and selected Path B (source-first
refactor in the `alawein` repo) over Path A (minimal patch) and Path C (retire the
monolith for surface-specific prompt files).

## §1 Goal and non-goals

**Goal.** Make the unified voice guide a doctrine-compliant, version-controlled
artifact in `alawein/docs/style/` with a single source of truth.

**Non-goals.**

- Not redesigning the four-block model (Block 1 universal core, Block 2
  design-defense register, Block 3 surface adjustments, Block 4 polish workflow).
- Not breaking `VOICE.md` consumers; the canonical Block 1 stays exactly where it
  is.
- Not retiring the monolith yet (Path C, deferred).
- Not extracting voice from `meshal.ai` (the site is supposed to follow VOICE.md;
  extracting voice from it is circular evidence).

they move to markdown source files in `docs/style/` and the assembler becomes pure
plumbing: concatenate, prepend frontmatter, demote H1s, write. The assembler holds
zero prose.

The output file carries a top-of-file warning comment:
`<!-- Generated by scripts/ops/build-voice-unified.py from docs/style/. Do not hand-edit. -->`

## §4 Project voice lives with the project

**Principle.** The central guide (`voice-unified.md`) covers *surfaces* —
software-doc, notebook, physics-paper, readme, claude-md, prompt-kit. Surfaces
are universal. *Projects* (Yosemite/Mercor, future jobs-projects engagements)
have their own tone constraints that belong with the project, not in the
workspace style contract.

**Effect on the assembler:**

- §20 (Yosemite) drops out entirely.
- The `[yosemite]` row drops from the §22 surface-tag table.
- No `--internal` flag. No dual outputs. The assembler has one mode, one output:
  `alawein/docs/style/voice-unified.md`.
- No pre-commit guard needed.
- The Desktop working copy (`Meshal-Alawein-Voice-Export.md`) is deleted after
  the cutover. The repo file is the only artifact.

**Effect on Yosemite's existing guide:**

- `alawein/jobs-projects/mercor/projects/project-yosemite/guides/voice_and_authorship.md`
  stays exactly where it is, untouched. It is now self-sufficient.
- The central guide's §22 gets a one-line footnote: *"Per-project voice contracts
  (e.g., Mercor/Yosemite) live in each project's own `guides/` directory and are
  not part of this guide."*

**Pattern for future jobs-projects.** When a new engagement needs project-specific
voice, it gets its own `guides/voice_and_authorship.md` (or equivalent) inside
`jobs-projects/<client>/<project>/`. The central guide does not grow per
engagement. If two or more engagements end up sharing voice constraints, factor
out a shared template at that point — not before.

## §5 Doctrine compliance

**Frontmatter for `voice-unified.md`** (matches the `VOICE.md` shape so existing
validators pass):

```yaml
---
type: generated
source: docs/style/VOICE.md, voice-software-register.md, voice-surfaces.md, voice-workflow.md
sync: scripts/ops/build-voice-unified.py
sla: regenerate on source change
authority: derived
audience: [contributors, agents, llm-system-prompts]
last_updated: <YYYY-MM-DD from git log of newest source>
---
```

`last_updated` is computed by the assembler as today's UTC date
(`datetime.now(timezone.utc).date()`). The earlier idea of using
`git log -1 --format=%cs` of source files turned out not to dodge the
`validate-doc-contract.sh` freshness check: same-day commits in the committer's
local timezone return the same %cs date, but the validator runs in UTC and
expects the field to advance. UTC date advances cleanly across CI runs and
matches what the validator's `today_iso` comparison expects.

**H1 strategy.** The output gets one H1 (the title `# Meshal Alawein — Voice
Guide`). Block headings demote: `# Block 1 ·` becomes `## Block 1 ·`. The current
`## §N` within blocks demotes to `### §N`. The four-block visual hierarchy
survives; the doctrine "single H1" rule passes.

VOICE.md currently has its own H1 (`# Voice and style contract`) that would
collide with `## Block 1 · Universal Core` if simply demoted. The assembler drops
that H1 when reading VOICE.md (the "Block 1 · Universal Core" header already
names the block). Other source files follow the same rule: any leading H1 that
names the block itself is dropped on read.

**voice-check linter exemption.** `voice-check` will flag the forbidden-register
list in §5 of Block 1 because it scans for those exact phrases as substrings.
Two options:

- *Option a (stopgap):* Add `docs/style/voice-unified.md` to a path-based skip
  list in `voice-check`.
- *Option b (target):* Add ignore-sentinel support to `voice-check`. Wrap each
  forbidden-register example with `<!-- voice-check:ignore-start -->` ...
  `<!-- voice-check:ignore-end -->`. Honor those sentinels in the scan loop.

Recommend option b. It is a general primitive: this spec, future post-mortems,
audit READMEs, and design-defense docs all need to quote forbidden phrases
legitimately. Path-based exemption (option a) only scales by adding more paths.
Sentinel implementation in `voice-check` is small (stateful flag in the existing
line-by-line scan).

The `voice-check` change is its own ticket. If sentinels do not land before this
refactor, ship with option a (path-based skip) and convert to sentinels later.
The spec lists option b as the target state and option a as the acceptable
interim.

## §6 Evidence enhancement scope

The original enhancement prompt asked for six additions. After the brainstorming
critique, the scope is:

**Keep, with sourcing discipline:**

- **§4 physics-paper register additions.** Pull 2-3 *direct quotes* from the PRM
  paper (`https://nano.eecs.berkeley.edu/publications/PRM_2025_Theory%20of%20Rippled%20Monolayers.pdf`)
  showing the claim-first / motivation-before-formalism pattern. Present as
  quotes with attribution, not as before/after pairs. The "before" version does
  not exist; do not invent it.
- **§8 concept-naming examples.** Pull 3-5 actual concept names from the PRM
  paper and the thesis (`https://escholarship.org/content/qt7vf7f1xh/qt7vf7f1xh.pdf`).
  Add them next to the existing eval_demo names. No before/after pairs.
- **§17 physics-paper expansions.** Pull 2-3 quoted passages from PRM showing
  the Feynman-register move (physical intuition before equation). Real quotes
  only, no fabricated bad version.

**Defer:**

- **§11a disagreement voice.** Published physics papers usually distinguish or
  refine prior work rather than openly disagree, so PRM and the thesis are weak
  evidence sources. The Yosemite "I disagree:" pattern (already in §11) is the
  strongest existing evidence. Defer §11a until there is a real corpus of
  disagreement writing (PR review comments, design-defense docs in eval_demo).
- **§26 source inventory.** Promote to a separate top-level `## Sources` section
  *only if* §4/§8/§17 actually cite PRM and the thesis. Decide after those
  enhancements are drafted.

**Cut:**

- **§0 voice fingerprint.** The "what the voice refuses to do (negative space)"
  framing produces aphoristic prose that the user has explicitly rejected
  (memory: `feedback_profile_no_tagline_quote.md`). The existing "How to use"
  header plus VOICE.md's own through-line ("Substance over performance...")
  already provides the orientation §0 was supposed to give.
- **`meshal.ai` web extraction.** The site is supposed to follow VOICE.md;
  extracting voice from it is circular evidence. If voice patterns do not match
  VOICE.md on the live site, that is a meshal-web bug, not a voice-guide gap.

**Net effect.** Three evidence enhancements (§4, §8, §17) instead of six, all
grounded in real text from PRM and the thesis, no fabricated bad examples, no
aphorism-generation risk.

## §7 Migration steps

Each step is independently committable and reversible.

1. **Extract Block 2/3/4 from the assembler into source files.** Create
   `voice-software-register.md`, `voice-surfaces.md` (without §20), and
   `voice-workflow.md` in `alawein/docs/style/`. Content is the existing Python
   triple-strings (BLOCK2, BLOCK3_SURFACE_ADJUSTMENTS plus the §15-16 from
   `voice-export-part4-tailoring.md`, BLOCK4) demoted by one heading level. Each
   gets the `VOICE.md` frontmatter shape.

2. **Move and rewrite the assembler.** Copy `_assemble_voice_export.py` to
   `alawein/scripts/ops/build-voice-unified.py`. Rewrite to: read four source
   files, demote H1 to H2 inline, compute `last_updated` from `git log`, prepend
   frontmatter, write to `alawein/docs/style/voice-unified.md`. No Yosemite read.
   No `--internal` flag. No dual-output logic.

3. **First clean build and doctrine validation.** Run the new assembler, then
   `bash scripts/doctrine/validate-doc-contract.sh --full` and
   `python scripts/doctrine/validate-doctrine.py .`. Fix any frontmatter or H1
   issues before going further. Voice-check failures on the forbidden-register
   examples are expected at this point.

4. **Add voice-check ignore-sentinel support** (or the path-based stopgap if
   sentinels are deferred to a separate ticket). Verify `voice-unified.md`
   passes voice-check.

5. **Evidence enhancements (§4, §8, §17).** Edit the *source files* — `VOICE.md`
   for §4 if those examples belong in Block 1, otherwise
   `voice-software-register.md` or `voice-surfaces.md`. Pull real quotes from
   PRM and the thesis. Regenerate. Validate.

6. **Delete stale Desktop artifacts.** Remove `Meshal-Alawein-Voice-Export.md`,
   `_assemble_voice_export.py`, `voice-export-part4-tailoring.md`, and
   `voice-guide-enhancement-prompt.md` from Desktop after confirming the repo
   build matches and is committed.

7. **Commit and push.** Single PR, conventional commit:
   `docs(style): unify voice guide with source-first build`. Body lists the four
   new source files and the new build script.

8. **Verify the raw GitHub URL resolves.**
   `https://raw.githubusercontent.com/alawein/alawein/main/docs/style/voice-unified.md`.
   GitHub raw CDN can cache up to 5 minutes; the first fetch may serve a stale
   404.

**Rollback for any step:** `git revert` the commit. Source files are independent
of each other; Block 2 changes do not break Block 1.

## §8 Open questions and future work

**Open questions (decide during implementation, not now):**

- **Where does the §4 physics-paper subsection live?** If it generalizes to all
  writing, it belongs in `VOICE.md` (Block 1). If it is specific to the
  `[physics-paper]` surface, it belongs in `voice-surfaces.md` next to §17.
  Likely the latter. Defer until the actual quotes are pulled from PRM.
- **`voice-check` ignore-sentinel feature** — separate spec or inline in this
  refactor's plan? Recommend separate, so this spec does not depend on a
  `voice-check` change landing first.
- **PRM quote selection** — which 2-3 passages from the paper best demonstrate
  claim-first / Feynman-register? Defer to implementation; flag in the plan as
  "pull 5 candidates, narrow to 2-3."

**Deferred to future specs:**

- **Path C (surface-specific prompt files).** The biggest reason to do this is
  paste-prompt token efficiency (~40KB → 5-8KB per surface). Worth doing once
  `voice-unified.md` is stable and being used in practice.
- **§11a disagreement voice.** Wait for a real corpus of disagreement writing.
- **§0 voice fingerprint.** Stays cut. Reopen only if there is a concrete use
  case (e.g., an LLM that needs a 200-token primer, not the full guide).
- **Per-project voice template.** If two or more `jobs-projects/` engagements end
  up needing a `guides/voice_and_authorship.md`, factor out a shared template.
  Not before.

**Out of scope for this spec entirely:**

- Modifying `meshal-web` or other consumers that currently reference VOICE.md.
- Adding new surfaces to the four-block model.
- Web fetching or source extraction from `meshal.ai`.
