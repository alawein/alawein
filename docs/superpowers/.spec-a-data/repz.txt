--- ARCHITECTURE ---

  hooks: useAASProtocol, useBloodworkResults, useDashboardData, etc. all use
  useQuery/useMutation with Supabase); React Context for auth/theme/site
  (src/context/); lightweight stores for auth session and notifications
  (src/stores/). The pattern is sound but has redundancy (see below).
Data fetching pattern: API calls live in domain hooks in src/hooks/ that
  wrap Supabase client calls. Services layer in src/services/ provides
  class-based AI services (WorkoutAI, NutritionAI, CoachingAI) with stub
  implementations — these are architectural placeholders, not live AI calls.
  The vision service (src/services/vision/) and TensorFlow/PoseNet deps suggest
  planned computer-vision features not yet active.
Component organization: domain-grouped within src/components/ (ai, auth,
  bloodwork, coaching, dashboard, charts, etc.) — good. BUT src/pages/ also
  has domain components co-located (src/pages/repz/components/), creating
  dual ownership zones for the same domain.
